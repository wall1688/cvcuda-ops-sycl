# CV-CUDA Remap 算子 CUDA→Intel SYCL 移植总结

## 一、代码理解

### 1.1 阅读范围
- src/cvcuda/priv/OpRemap.cu：核心 kernel（DoRemap）+ host 分发（RunRemap）+ 参数计算（GetRemapParams）
- 依赖头文件：
  - cuda_tools/InterpolationWrap.hpp：NEAREST/LINEAR/CUBIC 通用采样器
  - cuda_tools/BorderWrap.hpp：5 种 border 的 GetIndexWithBorder
  - cuda_tools/FullTensorWrap.hpp：张量包装
- 规格文件：include/cvcuda/OpRemap.h
- 变尺寸实现（参考）：src/cvcuda/priv/legacy/remap_var_shape.cu（类似结构）

### 1.2 算法要点
数学本质：对每个输出像素 (dx, dy, z)：
1. mapCoord = ((dx+dstOffset)*mapScaleX, (dy+dstOffset)*mapScaleY)
2. mapValue = sample_map(mapCoord, mapInterp) —— map 恒为 REPLICATE border，返回 float2
3. srcCoord = (dx*srcScaleX + mapValue.x*valScaleX + srcOffsetX, dy*srcScaleY + mapValue.y*valScaleY + srcOffsetY)
4. dst = sample_src(srcCoord, srcInterp, 用户 border)

与 Resize 的关键差异：

| 维度 | Resize | Remap |
|---|---|---|
| 采样次数 | 1 次（src） | 2 次（map + src，各自独立插值） |
| border | 写死的 clamp 边缘 | 5 种 border（CONSTANT/REPLICATE/REFLECT/WRAP/REFLECT101）通用处理 |
| 插值位置 | 固定缩放网格 | 任意 float 坐标 → 需通用 sampler |
| CUBIC 核 | Keys A=-0.75 | A=-0.5（OpenCV 核） |
| NEAREST | floor((d+0.5)*scale) | floor(c+0.5)（round-half-up） |
| map border | — | 恒为 REPLICATE，用户 border 只作用于 src |
| map 通道 | — | float2（C=2），与 src 的 T×C 分离 |

支持规格（来自 OpRemap.h）
- 布局：NHWC
- src dtype：F32 / U8，C=1/3/4（F32 仅 C1，U8 支持 C1/C3/C4）
- map dtype：float2（C=2）
- 插值：NEAREST / LINEAR / CUBIC（src 和 map 各自独立）
- border：CONSTANT / REPLICATE / REFLECT / WRAP / REFLECT101
- mapValueType：ABSOLUTE / ABSOLUTE_NORMALIZED / RELATIVE_NORMALIZED

本次实现范围：F32 / U8，C=1/3/4，4 种插值组合（srcInterp × mapInterp），5 种 border，3 种 mapValueType × alignCorners。

### 1.3 设计决策
- 完整移植 4 种插值组合 + var-shape
- 剥离 nvcv/cvcuda 框架（IOperator、TensorDataStridedCuda、cvcudaRemapSubmit 等不保留）
- interp/border 走 runtime switch（非模板），因组合空间 1620 种太大；只对 T×C（6 种）做模板
- 所有 scale/offset 在 host 计算传入，kernel 内无除法（绕开 Intel GPU FP 除法非 IEEE 问题）
- 测试采用自洽验证：CPU 参照与 GPU kernel 复用同一份 sample() 代码

## 二、CUDA → SYCL 核心映射

| CUDA (CV-CUDA) | SYCL（本移植） |
|---|---|
| `__global__ kernel<<<grid, block, 0, stream>>>(...)` | `q.parallel_for(range<3>{N,H,W}, [=](item<3> it){...})` |
| SrcWrapper/DstWrapper (FullTensorWrap) | `TensorView<T,C>`（自研最小 NHWC wrapper，USM 指针 + 形状） |
| `mapSrc[int3{x,y,z}]` | `m.read(x,y,z,mpix)`（float2 读） |
| `cuda::InterpolationWrap::operator[]` | `sample_impl<View>(v, fx, fy, z, W, H, interp, border, bv, out)` |
| `BorderWrap::GetIndexWithBorder` | `getIndexWithBorder(Border, c, s)`，逐字照抄整数模运算 |
| `cuda::SaturateCast<T>(v)` | `cvsycl::SaturateCast<T>(v)`（float 恒等，uint8_t 半偶舍入 + clamp [0,255]） |
| `GetCubicCoeffs` A=-0.5 | `getCubicCoeffs` A=-0.5（注意：与 Resize 的 A=-0.75 不同） |
| `cudaMalloc/Free + cudaMemcpyAsync` | `sycl::malloc_device<T>` / `sycl::free` + `q.copy<T>` |
| `cudaStream_t` / sync | `sycl::queue{gpu_selector_v, in_order}` / `q.wait()` |
| ImageBatchVarShapeWrap（var-shape） | `VarShapeBatch<T,C>`（pitched：rowPitch=maxW·C, imageStride=maxH·maxW·C） |
| 模板全展开（1620 种） | runtime switch + 只对 T×C 模板（6 种实例） |

## 四、编译调试

### 4.1 编译方式
Kernel 全部 header-only 模板化（remap.hpp + remap_varshape.hpp + common/*.hpp），test 直接 include，build.sh 只编译 3 个 test 二进制。

最终编译命令（build.sh 内）：
```

     -ffp-model=precise -fno-fast-math -fimf-precision=high \
     test/test_remap.cpp -o test/test_remap
icpx ... test/perf_remap.cpp -o test/perf_remap
icpx ... test/test_remap_varshape.cpp -o test/test_remap_varshape
```

### 4.2 修复的移植问题

| 问题 | 原因 | 修复 |
|---|---|---|
| 模板参数语法错误 | `T (*SrcFn](int, int, int)` 写错括号 | 改为 `T (*SrcFn)(int, int, int)` |
| var-shape 用例 FAIL（maxerr=1.0） | map 偏移用了 0.5，使 mapValue 正好落在 NEAREST 舍入边界 k+0.5；host/device FMA 差 1 ULP 即翻转 | 改用 0.25/0.75 远离边界 |
| 性能测量冷启动异常 | warmup 3 次不够，第一个配置被 JIT/冷 cache 拖累 | warmup 提到 5 次 |
| 性能计时非 best-of-N | `(t_end-t_start)/iters` 含核间间隔 | 改为逐核 profiling + wait + best-of-N |

重要教训：
- CUBIC A=-0.5（OpenCV Keys 核），不是 Resize 的 A=-0.75——混用会出错
- NEAREST = floor(c+0.5)，src 和 map 都是
- map border 恒为 REPLICATE，用户 border 只给 src
- 0.5 偏移是雷区：半整数在 NEAREST 边界上，FMA 差 1 ULP 就翻转；测试用 0.25/0.75
- 容器无 icpx，一律宿主机编译（与体素化/Resize 相同）

## 六、性能优化

### 6.1 优化方法：SYCL event profiling
在优化之前建立 per-kernel/per-场景计时能力，用于定位瓶颈和验证优化效果。
实现方式：
- perf_remap.cpp：SYCL queue 通过 enable_profiling 属性开启 event 时间戳记录
- 大图测试：1920×1080×4 batch，identity map + NEAREST map 隔离 src 插值开销
- warmup + best-of-20 取 device 时间（逐核 profiling + wait，最可复现）

### 6.2 Baseline 数据（Intel Arc [0xb080], opencl:gpu, best-of-20）

| config | ms | Gpix/s | GB/s |
|---|---|---|---|
| f32c1 NEAR REPL | 0.570 | 14.55 | 145.5 |
| f32c1 LIN REPL | 1.026 | 8.08 | 177.8 |
| f32c1 CUB REPL | 2.829 | 2.93 | 205.3 |
| f32c1 LIN REFL | 1.647 | 5.04 | 110.8 |
| f32c1 CUB REFL | 5.299 | 1.57 | 109.6 |
| u8c4 NEAR REPL | 0.646 | 12.84 | 128.4 |
| u8c4 LIN REPL | 1.324 | 6.27 | 137.9 |
| u8c4 CUB REPL | 3.197 | 2.59 | 181.6 |

热点定位：f32c1 CUB REFL 5.299ms（是 REPL 2.829ms 的 1.87×）。根因：getIndexWithBorder 中 REFLECT/WRAP/REFLECT101 的整数取模极贵，CUBIC 每像素 16 次取模。

### 6.3 优化：内部邻域快速路径
原理：绝大多数像素的整个插值邻域都在 src 范围内，此时 border 逻辑纯属浪费。在 sample_impl() 三个分支中加入快速路径：
```cpp
// NEAREST
if (ix >= 0 && ix < W && iy >= 0 && iy < H)
    v.read(ix, iy, z, out);
else
    fetch_impl(...);

// LINEAR
if (x1 >= 0 && x2 < W && y1 >= 0 && y2 < H) {
    v.read(x1, y1, z, p00); v.read(x2, y1, z, p10);
    v.read(x1, y2, z, p01); v.read(x2, y2, z, p11);
} else {
    fetch_impl(...);  // 4 次
}

// CUBIC
bool interior = (ix - 1 >= 0) && (ix + 2 < W) && (iy - 1 >= 0) && (iy + 2 < H);
if (interior)
    v.read(ix+cx, iy+cy, z, p);  // 16 次直读
else
    fetch_impl(...);  // 16 次走 border
```
安全性：getIndexWithBorder 对范围内索引在 5 种 border 下都是恒等变换，所以内部直读与走 border 路径 bit-identical。host/device 用同一个 sample_impl() 走同一决策 → maxerr 不变（38 用例回归全 PASS）。

### 6.4 A/B 对比结果

| config | base ms | opt ms | 加速比 |
|---|---|---|---|
| f32c1 NEAR REPL | 0.570 | 0.526 | 1.08× |
| f32c1 LIN REPL | 1.026 | 0.607 | 1.69× |
| f32c1 CUB REPL | 2.829 | 1.000 | 2.83× |
| f32c1 LIN REFL | 1.647 | 0.612 | 2.69× |
| f32c1 CUB REFL | 5.299 | 1.087 | 4.87× |
| u8c4 NEAR REPL | 0.646 | 0.584 | 1.11× |
| u8c4 LIN REPL | 1.324 | 0.786 | 1.68× |
| u8c4 CUB REPL | 3.197 | 1.787 | 1.79× |

关键结论：
- CUBIC REFLECT 4.87×（最大头）
- 优化后 REFL ≈ REPL（CUB: 1.087 vs 1.000，LIN: 0.612 vs 0.607）——border 类型不再拖垮性能，只有边缘像素走 border
- 复跑差 <0.3%，远超 ±10% 噪声阈值，结果可靠
- 38 用例 maxerr 完全不变（bit-identical 成立）

### 6.5 性能极限分析
当前瓶颈：已消除 border 取模热点。优化后性能受限于：
- 插值本身的访存量（NEAR 1 次读，LIN 4 次，CUB 16 次）——算法固有
- 小图场景 launch-bound（~0.1ms 固定开销）
- U8 多通道逐通道插值（比 F32 C1 慢）

未优化部分及原因：

| 场景 | 现状 | 原因 |
|---|---|---|
| NEAREST | 已快（0.53ms） | 1 次 fetch，无优化空间 |
| LINEAR | 已优化（REFL≈REPL） | 4 次读，快速路径已覆盖内部像素 |
| CUBIC | 已优化（REFL≈REPL，4.87×） | 16 次读，快速路径消除 border 取模热点 |
| 小图 | launch-bound | 固定开销主导，需 batch 融合，非单算子内可解 |

最终结论：在不改变 Remap 算法的前提下，主要瓶颈（REFLECT border 的整数取模）已通过内部快速路径解决。进一步提速需算子链融合或向量化 WritePack，属另一阶段工作。

### 6.6 CPU↔GPU speedup 对比

为回答"算子移植有没有做 CPU/GPU 对比验证"，给 perf_remap.cpp 加了 env 门控 `VOX_CPUTIME=1`（默认关，CUBIC 等 config 的 CPU gold 单线程较慢）的 CPU gold 参考计时。CPU 参考为朴素单线程实现（与正确性测试同一份 `remap_gold.hpp`，test 和 perf 共用、gold 单一来源），`speedup = cpu_ms / gpu_best_kernel_ms`（SYCL event profiling, best-of-20）。

运行：

```
ONEAPI_DEVICE_SELECTOR=opencl:gpu VOX_PROFILE=1 VOX_CPUTIME=1 ./test/perf_remap
```

实测（Intel Arc [0xb080]，1920×1080 ×4 batch，ABSOLUTE identity map，mapN=1 广播，1 run）：

| 配置 | gpu_best | cpu_ref | speedup |
|---|---|---|---|
| f32c1 NEAR REPL | 0.535ms | 63.37ms | 118.50× |
| f32c1 LIN  REPL | 0.645ms | 75.64ms | 117.22× |
| f32c1 CUB  REPL | 1.157ms | 177.00ms | 152.96× |
| f32c1 LIN  REFL | 0.761ms | 75.75ms | 99.58× |
| f32c1 CUB  REFL | 1.092ms | 178.48ms | 163.40× |
| u8c4  NEAR REPL | 0.693ms | 68.38ms | 98.64× |
| u8c4  LIN  REPL | 1.059ms | 225.43ms | 212.92× |
| u8c4  CUB  REPL | 1.748ms | 410.34ms | 234.73× |

说明：
- speedup 反映"GPU vs 朴素单线程 CPU"量级，**非优化基线**（公平的 GPU 自身性能对比见 6.2/6.4 的 baseline→opt）。
- Remap CPU gold 因 2 次采样（map + src）+ 通用 sampler + border 索引逐像素计算很慢，speedup 量级高（99×–235×），主要体现 GPU 并行对朴素单线程的压倒性优势，不意味 kernel 已达硬件极限。
- CPU 参考非 NVIDIA bit-diff（无 NVIDIA GPU），逻辑逐字照抄原版，局限见七章。

## 七、验证局限

| 局限 | 说明 |
|---|---|
| 自洽验证，非对齐验证 | 正确性测试用“同算法 CPU 参照”证明「SYCL kernel == 我的 CPU 实现」，不直接证明 == NVIDIA CV-CUDA 实际输出。若对原算法理解有误，CPU 和 GPU 会一起错，测试仍 PASS。缓解：关键逻辑（getRemapParams、GetCubicCoeffs A=-0.5、getIndexWithBorder 5 border、坐标公式）逐字照抄原版，理解错误风险集中在边缘细节。 |
| 未与 NVIDIA CV-CUDA 做 bit 级 diff | 真正的金标准是同一图像同一参数，对比 NVIDIA cvcudaRemap 与本 SYCL kernel 输出。未做（需 NVIDIA GPU + 编译 CV-CUDA，Intel Arc 跑不了 CUDA 版）。 |
| 合成数据 ≠ 真实图像极端情况 | 未用真实图片做端到端测试，可能存在未测到的边界（极大图整数溢出、特殊宽高比等）。 |
| U16/S16 dtype 未做 | 仅需增加 SaturateCast 特化即可，本次未覆盖。 |
| 未集成进 nvcv/cvcuda 框架 | 本移植为独立 SYCL 算子，非 drop-in 替换。 |
