# CV-CUDA NonMaximumSuppression 算子 SYCL 移植总结

> 本文件说明:做了什么、文件夹里是什么、怎么编译运行、正确性与性能如何。
> 更详尽的过程上下文见同目录 `claude_task_nms`。

## 一、算子移植

### 1. 代码理解
1. 阅读了 CV-CUDA 0.16.0 的 `src/cvcuda/priv/OpNonMaximumSuppression.cu` 及依赖(`OpNonMaximumSuppression.{hpp,h}`、`cuda_tools/TensorWrap.hpp`、测试 `TestOpNonMaximumSuppression.cpp` 中的 GoldNMS 参考)。
2. 算法:**全对(all-pairs)IoU 的 mask-NMS**,单 kernel,embarrassingly parallel(每个 box 一个线程)。遍历同 batch 所有其他 box,若 `IoU(X,Y) > iouThreshold` 且本 box 分数更低(或等分但面积更小)则丢弃。输出 per-box keep-mask(uint8,1=保留/0=丢弃)。**这是 mask 输出,不是 torchvision 的排序索引列表**——忠实 CV-CUDA 语义。
3. 数据:bboxes `short4`(x,y,w,h,左上角+宽高)`[batch, numBBoxes, 4]`;scores `float32` `[batch, numBBoxes]`;out `uint8` `[batch, numBBoxes]`。IoU/area 用整数坐标运算,仅末尾一次 float 除法。
4. 关键布局推导:`Tensor2DWrap::operator[](int2 c)` 内部调 `doGetPtr(c.y, c.x)`,即 CUDA 的 `coordX{bboxX, batchIdx}` 映射到 (dim0=batch, dim1=bbox),行主序连续 → 原生 USM 用 `b*numBBoxes + j` 索引,无转置。GoldNMS 用 `int2{x=batch, y=bbox}` 同向,语义一致。
5. tie-break 为严格 `<`(等分等面积互不丢弃),与 GoldNMS 完全一致;`break` 仅早退,与 Y 遍历顺序无关 → 结果确定。

### 2. CUDA → SYCL 核心映射
| CUDA | SYCL |
|---|---|
| `__global__ NonMaximumSuppression<<<grid,block,0,stream>>>`,grid.z=batch,threadIdx.x=bbox | `q.parallel_for(range<1>{S*N}, λ)`,内部分解 `b=idx/N, j=idx%N` |
| `cuda::Tensor2DWrap<short4>` `[coordX]` | 原生 USM `short*` + `b*numBBoxes + j` 索引(布局已推导为行主序连续) |
| `short4`(.x/.y/.z=宽/.w=高) | 本地 POD `struct short4{short x,y,z,w;}`(字段语义与 CUDA 一致,最小化翻译误差) |
| `cuda::max / cuda::min` | `sycl::max / sycl::min`(显式转 int 保证 OpenCL 后端可移植) |
| `atomicCAS` 等 | **无**——NMS 无原子、无 hash 表 |
| `cudaStream_t stream` | `sycl::queue*`(nullptr 用内部 queue,in-order + `VOX_PROFILE` 开 `enable_profiling`) |
| kernel 返回 void | `parallel_for` 返回 `sycl::event`,保存供 `last_profile()` 计时 |

### 4. 编译调试
1. kernel header-only(`nms.hpp`),test 直接 include,`build.sh` 只编 3 个 test 二进制,无需单独编 kernel。
2. 编译选项 `icpx -fsycl -std=c++17 -O2 -g -I. -ffp-model=precise -fno-fast-math -fimf-precision=high`(由 `build.sh` 编 3 个目标)。
3. 为 OpenCL 后端可移植性,`sycl::max/min` 显式转 int。
4. 入参校验用 `std::invalid_argument` 抛出(替代原项目 `Assertf/Asserts` 宏,使本算子自包含、不依赖外部 `common/check.hpp`)。

## 二、性能

### 1. 测试数据
NMS 无标准公开数据集,测试数据在程序内随机生成,模拟检测器输出:box 坐标整数 `pos∈[0,128)`、`wh∈[8,64]`、score `∈[0,1)`。默认规模 `S=4, N=5000`(共 20000 box),可参数化 `[N] [S] [runs]`。阈值取 `score_thresh=0.3, iou_thresh=0.45`(0.45 避开 1/2 整数比碰撞)。

### 2. 随机数据正确性
逐 box 比对 mask;对每个不一致 box,计算其与各 Y 的精确有理 IoU 到阈值的最小距离,若 < 1e-3 判定为 IoU 边界 case。
```
Running on: Intel(R) Graphics [0xb080]
CPU kept=1904 / 20000
mismatch=2 match=99.99% of which IoU-boundary(±0.001)=2 (100%)
[perf] wall best=1.285ms avg=2.734ms throughput=15.56 Mbox/s
=== REAL-DATA TEST PASSED ===
```
关键发现:
1. 20000 box 中仅 2 个 mask 不一致,一致率 99.99%。
2. 2 个不一致 **100% 为 IoU 恰在阈值的边界 case**(精确有理 IoU 到 0.45 的距离 < 1e-3)。
3. 根因:Intel Arc GPU 的 float 除法指令非 IEEE-754 正确舍入(基于倒数近似),`-ffp-model=precise` 无法改变。NMS 中 `iou = interArea/unionArea` 这一次 float 除法,在 `iou` 恰等于阈值时,GPU 倒数近似可能落到阈值另一侧,翻转 `> iouThreshold` 的判定。这是设备固有特性,不是移植 bug。
4. 说明:边界分类用的是自写的 `min_iou_dist_to_thresh`,存在循环论证风险;更严谨的做法是把 2 个 divergent box 的精确 IoU 值打印出来确认卡在阈值上——这是后续可补的独立校验。

### 3. 性能优化
NMS 是单 kernel、无原子、无 hash 表、无中段 host sync、无防御性 memset 可砍——**没有 opt 阶梯**。单 kernel 本身已是「无 sync、无原子」形态,overhead 极小。本轮仅建立 profiling 能力 + 性能极限分析,未做算法级优化(按「先 1:1 跑通再谈优化」原则)。

#### profiling 方法:SYCL event profiling
1. `nms.hpp`:queue 通过 `VOX_PROFILE` 环境变量 opt-in 开 `enable_profiling`,`forward()` 保存 kernel event,`last_profile()` 用 `get_profiling_info` 提取 start/end。
2. `test/test_nms_profile.cpp`:跑 forward → wait → 读 `last_profile`,输出 kernel best/avg ms + wall + overhead + 吞吐。

profiling 数据(S=4, N=5000, 20 runs,GPU warm):
```
----- NMS per-kernel (best) -----
nms_kernel   best=1.276ms avg=1.281ms (100%)
kernel sum   =1.276ms
wall         best=1.286ms avg=1.293ms
overhead     =0.0097ms (launch + wait)
throughput   kernel=15.67 Mbox/s  wall=15.55 Mbox/s
```
结论:**overhead 仅 0.0097ms**(kernel sum ≈ wall),证明单 kernel 无中段 host sync、无多余 launch/memset 开销。

#### 性能极限分析
当前特性:NMS 是 **O(N²) 访存,延迟受限**。
- 每个 box 线程需遍历同 batch 全部 N 个 box(读 bboxes + scores)→ 20000 线程 × 5000 box 读 ≈ 10⁸ 次随机读。
- 访存模式:boxY 按线性地址遍历(连续),但每线程独立遍历、跨线程无数据复用 → cache 命中率低,随机访问 cache miss 延迟主导。
- overhead 已极小(0.01ms),无「不可减 memset/launch 开销」可继续砍。

最终结论:在不改变 all-pairs 算法的前提下,当前 1.276ms / 15.6 Mbox/s 已是该实现的单 kernel 形态极限。进一步提速需**算法级改动**:shared-memory tiling 让同 block 线程共享 boxY 读取(降随机读放大)、或按 score 排序后早退、或分块——属另一阶段优化工作。

## 三、验证局限(诚实声明)
1. **自洽验证,非对齐验证**:正确性测试用的是我自己写的 CPU gold(移植自 CV-CUDA GoldNMS,与 `.cu` kernel 是两份独立实现),证明「SYCL kernel == 我的 CPU 参考」,**不直接证明 == NVIDIA CV-CUDA 实际输出**。若 kernel 与 gold 都从同一份源理解错,二者会一致地错,测试仍 PASS。缓解:kernel 体仅约 10 行,与原 `.cu` 逐行对照;gold 取自源仓独立参考。
2. **未与 NVIDIA CV-CUDA 做 bit 级 diff**:真正的金标准是同参数喂 NVIDIA `cvcudaNonMaximumSuppression` 和本 SYCL kernel 比输出。这一步没做(需 NVIDIA GPU + 编译 CV-CUDA,Intel Arc 跑不了 CUDA 版)。
3. **边界分类的循环论证风险**:real 测试用自写的 `min_iou_dist_to_thresh` 判定边界 case,存在循环论证风险;更严谨的做法是 dump 2 个 divergent box 的精确 IoU 确认卡在阈值上。
4. **FP 归因是推断**:Intel GPU 除法非 IEEE 舍入的归因来自 precedent + 边界距离检查,非单独证明的根因。

## 四、与其它算子移植的差异小结
| 维度 | 体素化 | NMS |
|---|---|---|
| kernel 数 | 3(build_hash / voxelization / reduce_mean) | 1 |
| 原子/hash 表 | 有(atomicCAS + 开放寻址) | 无 |
| 优化阶梯 | opt1-4,累计 3.5-3.8× | 无阶梯,单 kernel 已最简 |
| overhead | ~0.10-0.12ms(memset + 3 launch) | ~0.01ms(单 launch) |
| 瓶颈 | 随机 hash 访存,延迟受限 | O(N²) 随机 box 读,延迟受限 |
| FP 边界 | 点除法→体素 ±1 平移(0.04%) | IoU 除法→阈值翻转(0.01%) |
| 输出形态 | op 拥有输出缓冲 | 调用方传 out 指针(匹配 CV-CUDA API) |
| 算子来源 | NVIDIA Lidar_AI_Solution(MIT) | CV-CUDA(Apache-2.0) |

共性:都用 OpenCL 后端;FP 差异都归因到 Intel GPU 除法非 IEEE 舍入(设备固有,非 bug);性能极限都属「随机访存延迟受限,需算法级改动才能突破」。
