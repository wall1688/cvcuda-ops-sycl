# CV-CUDA Resize 算子 SYCL 移植总结

> 环境（Intel 开发板熟悉、icpx/opencl 后端等）已在体素化算子任务中完成，本任务直接复用，不再赘述。

## 一、Resize 算子移植

### 1. 代码理解

1. 阅读了 `priv/OpResize.cu`（4 种插值 kernel + host 分发）及依赖头文件（`cuda_tools/{SaturateCast,TensorWrap,InterpolationWrap,MathOps}.hpp`、`include/OpResize.h` 规格、`priv/legacy/resize_var_shape.cu` 变尺寸 kernel）。
2. 算法：4 种插值 ——
   - **NEAREST**：1 像素采样，`srcCoord=(dst+0.5)*scale`，floor + clamp。
   - **LINEAR**：2×2 双线性，`srcCoord=(dst+0.5)*scale-0.5`，边缘 wx=0/1 + clamp，CUDA 原版有 NIX+INTERSECT 读复用优化。
   - **CUBIC**：4×4 双三次，Keys 核 A=-0.75，`fx` 在 x 边缘置零（**非对称**，fy 不置零只 clamp），abs + SaturateCast。
   - **AREA**：盒均值，足迹 `[x·sx,(x+1)·sx)×[y·sy,(y+1)·sy)`（**无 +0.5 偏移**），xmin=ceil / xmax=floor，整数 scale 走简单均值，非整数走 9 项分数边缘，所有 scale 在 host 算好传入（kernel 无除法）。
3. 支持规格（来自 `OpResize.h`）：布局 NHWC/HWC，通道 1/3/4，dtype U8/U16/S16/F32。本次实现 **F32/U8 × C1/C3/C4**（U16/S16 加 SaturateCast 特化即可，未做）。
4. 决定完整移植 4 种插值 + var-shape，剥掉 nvcv/cvcuda 框架（IOperator/TensorDataStridedCuda/cvcudaResizeSubmit 全不要），只留 kernel 算法 + 自研最小 tensor wrapper。

### 2. CUDA → SYCL 核心映射

| CUDA (CV-CUDA) | SYCL (本移植) |
|---|---|
| `__global__ kernel<<<grid,block,0,stream>>>(...)` | `q.parallel_for(range<3>{N,H,W}, [=](item<3> it){...})` |
| `SrcWrapper/DstWrapper`（TensorWrapNHW） | `TensorView<T,C>`（自研最小 NHWC wrapper，USM 指针 + 形状） |
| `src[int3{x,y,z}]` / `dst.ptr(z,y)` | `s.read(x,y,z,pix[C])` / `d.write(x,y,z,pix[C])` |
| `cuda::SaturateCast<T>(v)` | `cvsycl::SaturateCast<T>(v)`（float 恒等，uint8_t 半偶舍入 + clamp [0,255]） |
| `cuda::min/max/floor/abs` | `sycl::min/max/floor/fabs` + `cvsycl::min/max/clamp` 别名 |
| `cuda::DropCast<2>` / `StaticCast<float>` | 直接 float 运算（C=1 简化） |
| `cudaMalloc/Free` + `cudaMemcpyAsync` | `sycl::malloc_device<T>(n,q)` / `sycl::free` + `q.copy<T>` |
| `cudaStream_t` / sync | `sycl::queue{gpu_selector_v, in_order}` / `q.wait()` |
| `NIX<T>` pack write（WritePack） | NIX dst 像素/线程 + INTERSECT 读复用（仅 U8 开） |
| `ImageBatchVarShapeWrap`（var-shape） | `VarShapeBatch<T,C>`（pitched：rowPitch=maxW·C，imageStride=maxH·maxW·C） |

### 4. 编译调试

1. kernel 全部 header-only 模板化（`resize.hpp` / `resize_varshape.hpp` 全模板 inline），test 直接 include，build.sh 只编 3 个 test 二进制，无需单独编 kernel。
2. 修了若干移植问题：
   - **C++ 模板特化顺序**：`SaturateCast` 的 `<float>`/`<uint8_t>` 全特化必须写在主模板声明之后，否则报 "no matching function for call to 'SaturateCast'"。调换顺序（主模板先声明）修复。
   - **负数左移 UB**：phase5b 初版写过 `int iPrevX = -1 << 30`，是未定义行为。改 `int iPrevX = 0` + `bool have` 守卫（只在 have=true 时读 iPrevX）。
   - **CUBIC 非对称边缘**：原版只对 `fx`（x 轴）在边缘置零，`fy`（y 轴）不置零只 clamp。必须照抄，否则输出不一致。
3. 容器无 icpx 编译器（同体素化任务），一律宿主机编译。
4. 最终编译命令（build.sh 内，编 3 个二进制）：
   ```
   icpx -fsycl -std=c++17 -O2 -g -I. \
        -ffp-model=precise -fno-fast-math -fimf-precision=high \
        test/test_resize.cpp -o test/test_resize
   icpx ... test/test_resize_profile.cpp -o test/test_resize_profile
   icpx ... test/test_resize_varshape.cpp -o test/test_resize_varshape
   ```

## 二、性能优化

### 1. 优化方法：SYCL event profiling

在优化之前，先建立了 per-kernel / per-场景计时能力，用于定位瓶颈和验证优化效果。

实现方式：
1. `test_resize_profile.cpp`：SYCL queue 通过 `enable_profiling` 属性开启 event 时间戳记录。
2. 大图测试场景：F32 C1 + U8 C3 + U8 C1，输入 1920×1080，上采样到 3840×2160 / 下采样到 960×540。
3. warmup + best-of-20 取 device 时间，输出 best ms + dst Gpix/s + GB/s。

baseline 数据（Intel Arc [0xb080]，opencl:gpu，best-of-20）：
```
[F32 C1 downscale 1920x1080->960x540]   NEAR 0.114 / LIN 0.125 / CUB 0.122 / AREA 0.123 ms
[F32 C1 upscale  ->3840x2160]           NEAR 0.323 / LIN 0.358 / CUB 0.502 ms
[U8 C3 downscale ->960x540]             NEAR 0.115 / LIN 0.151 / AREA 0.143 ms
[U8 C3 upscale   ->3840x2160]           NEAR 1.279 / LIN 3.267 ms
[U8 C1 upscale   ->3840x2160]           NEAR 0.293 / LIN 0.346 ms
```

结论：
1. **小图 launch-bound**：960×540 输出 ~0.1ms，4 种插值差异极小，固定 launch 开销主导。
2. **U8 C3 上采样 4K LINEAR 3.267ms 异常慢**（NEAREST 1.279ms 的 2.5×）—— 3 字节像素非对齐访问，linear 多次按通道循环 read，是主要可优化点。
3. F32 C1 大图已较快（~24 Gpix/s），优化空间小。

### 2. LINEAR NIX+INTERSECT 读复用优化

移植 CUDA 原版的 `LinearInterpolatePack<INTERSECT=true>`。

**原理**：每线程处理 NIX 个连续 dst 像素，相邻 dst 映射到同一 src 时复用 2×2 邻域，src x+1 时只增量读 2 个像素（p00←p10、p01←p11 平移复用），减少重复访存 + 减少 launch 线程数。

**修改**：
- `resize.hpp` 的 `linear_resize` 改为 NIX 像素/线程 + INTERSECT 复用（`iPrevX`/`have` 守卫增量读）。
- `kNIX` 门控：`kNIX = (sizeof(T)==1) ? (kPackBytes/(sizeof(T)*C)) : 1`。C=1→16，C=3→4，C=4→4；**F32 走 NIX=1 简单路径**。

**正确性**：优化后 25 用例 bit-一致（U8 LINEAR 在 ±1 容差内，实测 maxerr=0）。

**性能（baseline → 优化后）**：

| 场景 | baseline | 优化后 | 变化 |
|---|---|---|---|
| F32 C1 上采样 4K LINEAR | 0.350ms / 23.7 Gpix/s | 0.358ms / 23.2 | 持平（门控保护） |
| **U8 C3 上采样 4K LINEAR** | 3.267ms / 2.5 Gpix/s | **1.790ms / 4.6** | **1.83×** ⬆️ |
| U8 C3 下采样 LINEAR | 0.151ms | 0.103ms | 1.47× ⬆️（反超 NEAREST） |
| U8 C1 上采样 4K LINEAR | — | 0.346ms / 24.0 Gpix/s | NIX=16 表现好 |

### 3. kNIX 门控的关键洞察

**根因**：初版对 F32 也开了 NIX=4，结果 F32 C1 上采样 LINEAR **回退 8%**。

**分析**：F32 是 4 字节对齐读，本就便宜；NIX 减线程数反损并行度（Arc GPU 对并行度敏感）。U8 是 1 字节，多通道非对齐访问贵，NIX+INTERSECT 的读复用收益大于并行度损失。

**修改**：`kNIX = (sizeof(T)==1) ? 公式 : 1` —— 只对 U8 开 NIX+INTERSECT，F32 走 NIX=1 简单路径。

**结果**：F32 持平（不再回退），U8 获 1.83× 加速。**移植 CUDA 优化时不能盲目照搬，要按本设备特性重新判断适用边界。**

### 4. 性能极限分析

当前状态：LINEAR（最常用插值）的 U8 多通道场景已优化到位。

**未优化的部分及原因**：
1. **NEAREST**：已快（U8 C3 上采样 1.279ms），1 像素采样无读复用空间。
2. **CUBIC**：4×4 采样理论上可做 NIX，但用得少（CUBIC 上采样 4K F32 0.502ms），收益小，未做。
3. **AREA**：scale 全 host 算好，kernel 无除法，已无 FP 瓶颈；盒均值访存模式与 LINEAR 不同，NIX 不直接适用。
4. **小图 launch-bound**：~0.1ms 固定开销主导，要更快需 batch 融合 / 算子链，非单算子内可解。

**最终结论**：在不改变插值算法的前提下，LINEAR 的主要瓶颈（U8 C3 多通道非对齐访存）已通过 NIX+INTERSECT 解决（1.83×）。进一步提速需算子链融合或向量化 WritePack，属另一阶段工作。

## 三、验证局限（诚实声明）

1. **自洽验证，非对齐验证**：正确性测试用的是我自己写的「同算法 CPU 参考」，证明「SYCL kernel == 我的 CPU 实现」，**不直接证明 == NVIDIA CV-CUDA 实际输出**。若我对原算法某处理解错，CPU 和 GPU 会一起错，测试仍 PASS。缓解：关键逻辑（坐标公式、`get_cubic_coeffs` A=-0.75、AREA 的 ceil/floor 边缘、CUBIC 非对称 fx 置零、SaturateCast 半偶舍入）**逐字照抄原版**，非重写，理解错风险集中在边缘细节。
2. **未与 NVIDIA CV-CUDA 做 bit 级 diff**：真正的金标准是同图同参数喂 NVIDIA `cvcudaResize` 和本 SYCL kernel 比输出。这一步没做（需 NVIDIA GPU + 编译 CV-CUDA，Intel Arc 跑不了 CUDA 版）。
3. **合成数据 ≠ 真实图像极端情况**：没用真实图片做端到端测试，可能存在未测到的边界（极大图整数溢出、特殊宽高比等）。
4. **AREA var-shape 未移植**：原版用另一套 OpenCV-aligned 实现 `resize_area_ocv_align` + BorderVarShapeWrap，复杂少用，标注为限制。
5. **U16/S16 dtype 未做**：加 SaturateCast 特化即可。
6. **未集成进 nvcv/cvcuda 框架**：独立 SYCL 算子，非 drop-in 替换。
